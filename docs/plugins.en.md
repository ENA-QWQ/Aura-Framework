## 1. Overview

A plugin is an independent directory placed under the framework root `plugins/` folder (at the same level as `src/`).  
Each plugin must provide a `manifest.json` file. It may also provide a Node.js entry (for build‑time hooks), a browser entry (client‑side scripts), and stylesheets.

Plugins are enabled by listing their `name` in the `plugins` array of `aura.config.ts`.

---

## 2. Directory Structure

```
plugins/
  my-plugin/
    manifest.json         # Required
    node.js               # Optional, Node.js hooks
    browser.js            # Optional, browser script (ES Module)
    styles.css            # Optional, stylesheet
    ...                   # Other assets
```

Aura scans this directory at the start of the build and loads all plugins that appear in the user‑configured `plugins` list.

## 3. Manifest File (manifest.json)

| Field          | Type                               | Required | Description |
|----------------|------------------------------------|----------|-------------|
| `name`         | string                             | Yes      | Unique identifier; must match the folder name and be referenced in `aura.config.ts`. |
| `version`      | string                             | No       | Plugin version. |
| `enforce`      | `"pre"` \| `"normal"` \| `"post"`  | No       | Execution phase order; defaults to `"normal"`. |
| `dependencies` | string[]                           | No       | List of other plugin names this plugin depends on; ensures these plugins run before the current one. |
| `schema`       | object                             | No       | Schema definition used to validate data stored via `DataStore`. Keys are data keys, values are `SchemaDefinition`. |
| `entry`        | object                             | No       | Entry file path configuration. |
| `entry.node`   | string                             | No       | Node.js entry file path (relative to the plugin directory). This file should export a `PluginHooks` object or a default export containing hook methods. |
| `entry.browser`| string                             | No       | Browser script path (injected into `<body>` as an external `<script type="module">`). |
| `entry.styles` | string                             | No       | Stylesheet path (injected into `<head>` as an inline `<style>`). |

### SchemaDefinition Type

```typescript
interface SchemaDefinition {
  type: 'string' | 'number' | 'boolean' | 'object' | 'array';
  items?: SchemaDefinition;                      // Used when type is 'array'
  properties?: Record<string, SchemaDefinition>; // Used when type is 'object'
  required?: boolean;                            // Defaults to false
}
```

Example manifest with schema:

```json
{
  "name": "blog",
  "version": "1.0.0",
  "enforce": "normal",
  "dependencies": [],
  "schema": {
    "posts": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "title": { "type": "string", "required": true },
          "date": { "type": "string" }
        }
      }
    }
  },
  "entry": {
    "node": "node.js",
    "browser": "browser.js",
    "styles": "styles.css"
  }
}
```

## 4. Plugin Hooks

All hooks are defined in the `PluginHooks` interface. They can be synchronous or asynchronous (returning a Promise).  
Hooks are called in order determined by `enforce` and `dependencies` (topological sorting).

### Hook Types

- **Trigger** – Side‑effect only; receives or returns no data payload.
- **Waterfall** – Receives the return value of the previous plugin and returns a new value; the final value becomes the result of that stage.

### Hook List

| Hook Name              | Type       | Parameters                                                         | Return Value       | Description |
|------------------------|------------|--------------------------------------------------------------------|--------------------|-------------|
| `buildStart`           | Trigger    | `(ctx: AuraContext) => void`                                       | -                  | Called once when the build starts; suitable for initialisation. |
| `loadContent`          | Waterfall  | `(collections: Collection[], ctx: AuraContext) => Collection[]`    | `Collection[]`     | Load or generate content collections. The returned array **replaces** `ctx.collections`. |
| `fetchData`            | Trigger    | `(ctx: AuraContext) => Record<string, any>`                        | Data object        | Fetch arbitrary data. The returned object is automatically stored in `ctx.data` under the plugin name as namespace (one property per key). You may also call `ctx.data.set()` manually. |
| `transformCollections` | Waterfall  | `(collections: Collection[], ctx: AuraContext) => Collection[]`    | `Collection[]`     | Transform collections after `loadContent`. The returned array **replaces** the collection list. |
| `generateRoutes`       | Waterfall  | `(routes: Route[], ctx: AuraContext) => Route[]`                   | `Route[]`          | Generate or modify routes. The returned array **replaces** `ctx.routes`. |
| `transformHtml`        | Waterfall  | `(html: string, ctx: AuraContext, route: Route) => string`         | `string`           | Transform the final HTML of each route before writing to disk. |
| `buildEnd`             | Trigger    | `(ctx: AuraContext) => void`                                       | -                  | Called after all files are written; suitable for cleanup or statistics. |

**Important:** `loadContent`, `transformCollections`, and `generateRoutes` are **replacement** hooks, not merging hooks. The returned value completely replaces the current state.

## 5. Context (AuraContext)

Every hook receives the `ctx` object, which contains all build state and utilities.

```typescript
interface AuraContext {
  config: ResolvedConfig;            // Final resolved configuration
  collections: Collection[];         // Current collection list
  routes: Route[];                   // Current route list
  data: DataStore;                   // Cross‑plugin data store
  assets: AssetManager;              // Asset manager
  components: Map<string, Function>; // Component registry (for templates)
}
```

### DataStore

Used to share structured data between plugins, with optional schema validation.

```typescript
class DataStore {
  set(namespace: string, key: string, value: any, schema?: SchemaDefinition): void;
  get(namespace: string, key: string): any;
  getFlat(): Record<string, any>; // Returns { namespace: { key: value, ... }, ... }
}
```

- `namespace` – It is recommended to use your plugin name to avoid collisions.
- If `schema` is provided, it is validated immediately; an error is thrown on failure.

In `fetchData`, if you return an object, Aura automatically calls `ctx.data.set(pluginName, key, value)` for each key, using the schema defined in the manifest (if any). You may also call `set()` manually.

### AssetManager

Used to add CSS/JS assets that will be injected into the final HTML.

```typescript
class AssetManager {
  add(type: 'css' | 'js', content: string, inject: 'head' | 'body', filename?: string, pluginName?: string): void;
  getAssets(): AssetDescriptor[];
}
```

- `type` – Asset type.
- `content` – Raw code or file content.
- `inject` – Injection location (`head` or `body`).
- `filename` + `pluginName` – If provided, the asset is written to `outDir/assets/plugins/pluginName/filename` and linked as an external resource; otherwise it is inlined.

**Note:** The framework automatically handles `entry.browser` and `entry.styles` – they are internally added via `assets.add()`. You only need to call `assets.add()` for dynamically generated assets.

## 6. Collections and Routes

### Collection

```typescript
interface Collection {
  name: string;          // e.g. "posts"
  items: CollectionItem[];
}

interface CollectionItem {
  id: string;            // Unique identifier
  slug: string;          // URL‑friendly identifier
  collection: string;    // Name of the containing collection
  metadata: Record<string, any>; // Front‑matter or custom fields
  content: string;       // Raw body (Markdown, HTML, etc.)
}
```

Collections are the primary way to organise content. Typically you generate them in `loadContent` or `transformCollections`.

### Route

```typescript
interface Route {
  path: string;          // e.g. "/posts/hello-world"
  template: string;      // Template filename (without extension) under the theme's `templates/` directory
  modules: string[];     // Reserved for future use
  collection?: string;   // Optional – you may use it to associate a collection, but the core renderer does not use it
  pageType?: string;     // Optional – can be used for custom logic
  data: Record<string, any>; // Data passed to the template
}
```

Routes are generated in `generateRoutes`. They are then rendered by the theme's template engine.  
For each route, the framework writes `outDir/path/index.html` (or `outDir/index.html` if `path === '/'`).

## 7. Complete Example: Blog Plugin

This plugin reads Markdown files from `src/posts/`, creates a `"posts"` collection, generates a list route and detail routes for each post, and stores post data in `DataStore`.

### Directory Structure

```
plugins/blog/
  manifest.json
  node.js
```

### manifest.json

```json
{
  "name": "blog",
  "version": "1.0.0",
  "enforce": "normal",
  "schema": {
    "posts": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "title": { "type": "string", "required": true },
          "slug": { "type": "string", "required": true },
          "date": { "type": "string" }
        }
      }
    }
  },
  "entry": {
    "node": "node.js"
  }
}
```

### node.js (Hook Implementation)

```typescript
import { readdir, readFile } from 'fs/promises';
import { join } from 'path';
import matter from 'gray-matter'; // You need to install this package

export default {
  async loadContent(collections, ctx) {
    const postsDir = join(ctx.config.srcDir, 'posts');
    let files = [];
    try {
      files = await readdir(postsDir);
    } catch {
      return []; // No posts directory, return empty
    }

    const items = [];
    for (const file of files) {
      if (!file.endsWith('.md')) continue;
      const raw = await readFile(join(postsDir, file), 'utf-8');
      const { data, content } = matter(raw);
      const slug = data.slug || file.replace(/\.md$/, '');
      items.push({
        id: file,
        slug,
        collection: 'posts',
        metadata: data,
        content
      });
    }

    // Return a single collection named "posts"
    return [{ name: 'posts', items }];
  },

  // Store post data in DataStore for use by other plugins or templates
  async fetchData(ctx) {
    const postsCollection = ctx.collections.find(c => c.name === 'posts');
    if (!postsCollection) return {};
    // Return an object that will be stored under the "blog" namespace
    return { posts: postsCollection.items };
  },

  async generateRoutes(routes, ctx) {
    const posts = ctx.collections.find(c => c.name === 'posts')?.items || [];

    // Preserve existing routes (if any) and add new ones
    const newRoutes = [
      ...routes,
      {
        path: '/posts',
        template: 'post-list',   // Template file: post-list.ts
        modules: [],
        data: { posts }          // Data passed to the template
      },
      ...posts.map(post => ({
        path: `/posts/${post.slug}`,
        template: 'post-detail', // Template file: post-detail.ts
        modules: [],
        data: { post }
      }))
    ];

    return newRoutes;
  }
};
```

### Data Flow Explanation

1. `loadContent` reads files and returns a collection.
2. The collection is stored in `ctx.collections`.
3. `fetchData` reads that collection and stores it in `ctx.data` under `blog.posts`.
4. `generateRoutes` uses `ctx.collections` to build routes and passes post data directly into each route's `data` field.
5. Theme templates (e.g. `post-list.ts` and `post-detail.ts`) receive the `data` object as a parameter during rendering.

### Theme Template Example

In the theme's `templates/post-list.ts`:

```typescript
export default function render(ctx, route, data) {
  // data is what you passed in route.data
  const posts = data.posts || [];
  return `
    <ul>
      ${posts.map(p => `<li><a href="/posts/${p.slug}">${p.metadata.title}</a></li>`).join('')}
    </ul>
  `;
}
```

In `templates/post-detail.ts`:

```typescript
export default function render(ctx, route, data) {
  const post = data.post;
  return `
    <h1>${post.metadata.title}</h1>
    <div>${post.content}</div>
  `;
}
```

## 8. Dynamic Routes and Slug Handling

The core framework does not provide a built‑in dynamic routing mechanism. You need to explicitly generate routes in `generateRoutes` as shown above.  
To support pagination, tag archives, etc., simply generate multiple route objects in the same way, each with its own `data`.

Example: generate a route for each tag:

```typescript
const tags = new Set();
posts.forEach(p => (p.metadata.tags || []).forEach(t => tags.add(t)));
const tagRoutes = [...tags].map(tag => ({
  path: `/tags/${tag}`,
  template: 'tag',
  data: { tag, posts: posts.filter(p => (p.metadata.tags || []).includes(tag)) }
}));
```

## 9. Error Handling and Debugging

- If a hook throws an error, Aura catches it, logs the plugin name and error message, and continues the build (unless it is a fatal error, e.g. circular plugin dependencies).
- It is recommended to use try/catch around file I/O and external API calls, and log meaningful messages with `console.warn` or `console.error`.
- To inspect what data is available, you can temporarily print `ctx.data.getFlat()` or `ctx.collections` inside a hook.

---

## 10. Inter‑Plugin Communication

- Use `DataStore` to share data between plugins. For example, one plugin can store configuration: `ctx.data.set('myplugin', 'config', {...})`, and another can retrieve it via `ctx.data.get('myplugin', 'config')`.
- Use consistent namespaces (preferably plugin names) to avoid accidental overwrites.

---

## 11. Browser‑Side Scripts and Styles

- `entry.browser` is injected as an external `<script type="module">` pointing to `assets/plugins/<pluginName>/<filename>`.
- `entry.styles` is injected as an inline `<style>` into `<head>`.
- If you need to add dynamic client‑side code, you can call `ctx.assets.add('js', code, 'body', 'dynamic.js', 'myplugin')` from a hook (e.g. `transformHtml` or `buildEnd`).
- The framework injects a global event bus `window.AuraBus` into every page. You can use it to communicate between different browser scripts.

---

## 12. Execution Order (Hooks + enforce + dependencies)

1. Plugins are first sorted by `enforce` (pre → normal → post), then topologically sorted by `dependencies`.
2. For each hook phase, all plugins that implement that hook are called in the sorted order.
3. For waterfall hooks, the output of the previous plugin becomes the input of the next.

---

## 13. Important Caveats and Pitfalls

- `loadContent`, `transformCollections`, and `generateRoutes` **replace** the entire array, not merge. If you want to preserve previous values, you must explicitly copy them (e.g. `[...previous, ...new]`).
- `fetchData` expects to return a plain object. If it returns `undefined`, no data is stored.
- The `collection` field on a route is not used by the core renderer – it is only for informational purposes. You must pass the actual data inside `data`.
- Theme templates receive the full `ctx`, the `route` object, and the `data` object from the route. They can also access other data via `ctx.data.getFlat()`.

---

## 14. Complete Plugin Lifecycle

1. `discoverAndLoadPlugins` reads all manifest files and loads Node entries.
2. Plugins are sorted.
3. `buildStart` hooks are called.
4. In sequence: `loadContent` → `transformCollections` → `fetchData` → `generateRoutes`.
5. Asset files specified by `entry.browser` and `entry.styles` are copied and registered.
6. Theme styles are loaded.
7. For each route:
    - Render template → layout → inject assets via `transformHtml`.
    - Write HTML file.
8. `buildEnd` hooks are called.
